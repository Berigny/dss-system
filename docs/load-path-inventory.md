# KSR Runtime Load-Path Inventory

This document enumerates every `semantic_registry.yaml` load path in the DSS monorepo and classifies each reference according to the core-only graceful-degradation decision (D1).

## Classification legend

- **core-safe**: Reads fields that are present in `ksr-core`; no steward data required.
- **graceful-noop**: Expects steward-only fields; degrades to empty/None/log notice when only `ksr-core` is present.
- **pack-overlay**: Reads `ksr-pack-domains` or `ksr-pack-steward`; requires a deploy-time overlay.
- **private-artifact**: Requires the encrypted envelope `.enc` or the full source YAML; build/test/steward tooling only.

## Runtime summary

When the public repo loads `ksr-core` as `apps/backend/backend/kernel/semantic_registry.yaml`:

- Kernel constants generate correctly.
- Base foundation builds a valid record; cross-domain domains are empty but the registry structure is present.
- Encode/decode tools operate on engineering primes and coordinates.
- Steward-only lattice enrichment modules (`coord_enrichment.py`, `reverse_parser.py`, `output_formatter.py`, `embeddings.py`) are excluded from public esoteric-content scans and are only meaningful when the steward pack is present.
